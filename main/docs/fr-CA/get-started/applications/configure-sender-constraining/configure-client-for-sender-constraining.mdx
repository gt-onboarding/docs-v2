<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La restriction de l’expéditeur appliquée aux jetons à l’aide de Demonstrating Proof-of-Possession (DPoP) est actuellement en accès anticipé (Early Access). Pour demander l’accès à cette fonctionnalité, communiquez avec votre représentant Auth0.
</Callout>

[La restriction de l’expéditeur](/docs/fr-CA/secure/sender-constraining) est un mécanisme de sécurité <Tooltip tip="" cta="Afficher le glossaire" href="/docs/fr-CA/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> et <Tooltip tip="" cta="Afficher le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) qui lie de manière cryptographique les jetons d’accès et les <Tooltip tip="" cta="Afficher le glossaire" href="/docs/fr-CA/glossary?term=refresh+tokens">Jetons d’actualisation</Tooltip> à l’application cliente spécifique qui les a obtenus, ce qui empêche le vol et l’utilisation abusive de jetons.

Auth0 prend en charge la restriction de l’expéditeur mTLS et Demonstrating Proof-of-Possession (DPoP). Si vous activez la restriction de l’expéditeur pour une application cliente, vous devez également l’appliquer au <Tooltip tip="" cta="Afficher le glossaire" href="/docs/fr-CA/glossary?term=resource+server">serveur de ressources</Tooltip> auquel vous effectuez des appels d’API.

Pour configurer la restriction de l’expéditeur dans Auth0, vous devez :

* [Configurer la restriction de l’expéditeur pour une application cliente](#configure-sender-constraining-for-a-client-application).
* [Configurer la restriction de l’expéditeur pour un serveur de ressources](#configure-sender-constraining-for-a-resource-server).

<div id="how-it-works">
  ## Comment ça fonctionne
</div>

Les <Tooltip tip="" cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Access+tokens">jetons d’accès</Tooltip> sont soumis à une contrainte d’expéditeur dans Auth0 selon la façon dont vous configurez la contrainte d’expéditeur pour votre application cliente et votre serveur de ressources :

1. **Audience demandée :** Dans une requête de jeton, le fait que l’audience demandée soit uniquement `/userinfo`, soit uniquement destinée à être utilisée avec le point de terminaison [`/userinfo`](https://auth0.com/docs/api/authentication/user-profile/get-user-info), ou soit une API personnalisée qui peut inclure `/userinfo` lorsque vous demandez également la portée openid, a une incidence sur le fait que les jetons d’accès soient soumis ou non à une contrainte d’expéditeur.
2. **Application cliente :** Le fait que vous définissiez la contrainte d’expéditeur comme `required` au niveau de l’application cliente.
3. **Serveur de ressources :** Le fait que vous configuriez la contrainte d’expéditeur pour le serveur de ressources :

   1. `none` : Vous n’avez pas configuré de contrainte d’expéditeur pour le serveur de ressources.
   2. `allowed` : Vous avez configuré une contrainte d’expéditeur pour le serveur de ressources en définissant une méthode de contrainte d’expéditeur.
   3. `required` : Vous avez configuré la contrainte d’expéditeur comme requise pour le serveur de ressources, ce qui signifie que les jetons d’accès doivent être associés à une contrainte d’expéditeur à une application donnée. Nécessite une méthode de contrainte d’expéditeur.

* **Preuve de possession :** Le fait que l’application cliente ait envoyé ou non une assertion de preuve de possession dans la requête de jeton :

  * Contrainte d’expéditeur mTLS : La preuve de possession est démontrée par la présentation réussie par le client d’une clé privée spécifique (associée à un certificat client) pendant la négociation TLS.
  * DPoP : La preuve de possession est obtenue lorsque le client crée un JWT de preuve DPoP en le signant cryptographiquement avec sa clé privée et en incluant le JWT de preuve `DPoP` dans l’en-tête HTTP DPoP de chaque requête où le jeton d’accès associé est utilisé.

<div id="configure-sender-constraining-for-a-client-application">
  ## Configurer la contrainte d’émetteur pour une application cliente
</div>

Lorsque vous exigez la contrainte d’émetteur pour une application cliente, les jetons d’accès sont limités à cette application. Auth0 vérifie les requêtes pour s’assurer que seule l’application qui a demandé le jeton peut l’utiliser pour accéder à la ressource associée.

Une fois que vous avez configuré une application cliente pour exiger la contrainte d’émetteur, vous pouvez définir la méthode de contrainte d’émetteur, soit mTLS ou DPoP, lorsque vous configurez votre serveur de ressources.

Vous pouvez configurer la contrainte d’émetteur pour une application cliente avec le <Tooltip tip="" cta="Afficher le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> ou la <Tooltip tip="" cta="Afficher le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>.

<Tabs>
  <Tab title="Auth0 Dashboard">
    1. Accédez à [Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications). Sélectionnez l’application que vous voulez configurer.
    2. Sous **Settings**, faites défiler jusqu’à **Token Sender-Constraining**.
    3. Activez **Require Sender Constraining**. Désactivez cette option pour supprimer l’exigence de Sender Constraining pour l’application.

    <Frame>![](/docs/images/cdy7uua7fh8z/3BoGW9NEYKPBkfk2tPBL7l/0f884ac3f64f17b23fd6a0e9436bb472/Screenshot_2024-07-23_at_1.30.19_PM.png)</Frame>
  </Tab>

  <Tab title="Management API">
    Pour configurer un client pour la contrainte d’émetteur (Sender Constraining), utilisez la [Management API](https://auth0.com/docs/api/management/v2).

    Pour exiger la contrainte d’émetteur pour un client, envoyez une requête PATCH pour [mettre à jour les paramètres du client](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id). Définissez le paramètre `require_proof_of_possession` sur `true`.

    ```json lines
    "require_proof_of_possession": true
    ```

    Pour supprimer l’exigence de contrainte d’émetteur, définissez le paramètre `require_proof_of_possession` sur `false`.

    ```json lines
    "require_proof_of_possession": false
    ```
  </Tab>
</Tabs>

<div id="configure-sender-constraining-for-a-resource-server">
  ## Configurer la restriction liée à l’émetteur pour un serveur de ressources
</div>

Les jetons d’accès émis par Auth0 peuvent être restreints à l’émetteur (c.-à-d. l’application cliente) qui doit accéder aux API d’un serveur de ressources.

Vous pouvez configurer la restriction liée à l’émetteur pour un serveur de ressources dans Auth0 Dashboard ou à l’aide de la Management API.

<Tabs>
  <Tab title="Auth0 Dashboard">
    Pour activer Token Binding ou la restriction liée à l’émetteur, configurez les **API Settings** de votre API.

    1. Accédez à [Auth0 Dashboard &gt; Applications &gt; APIs](https://manage.auth0.com/#/apis).
    2. Sélectionnez l’API que vous voulez configurer.
    3. Sous l’onglet **Settings**, trouvez la section **Token Sender-Constraining**.
    4. Configurez les éléments suivants :

       1. Sender Constraining Method :

          1. **None :** N’activez pas de méthode de restriction liée à l’émetteur pour votre serveur de ressources.
          2. **mTLS :** Activez mTLS comme méthode de restriction liée à l’émetteur pour votre serveur de ressources.
          3. **DPoP :** Activez DPoP comme méthode de restriction liée à l’émetteur pour votre serveur de ressources.

    B. Activez **Require Token Sender Constraining**. Tous les jetons d’accès émis pour une application pour cette API seront restreints à cette application.

    <Frame>![Auth0 Dashboard > APIs > Settings > Token binding](/docs/images/cdy7uua7fh8z/3Dv98iZosdpJMcZXfXyurn/823fa368ab874218fbb737aacac9b262/Screenshot_2025-07-28_at_3.59.01_PM.png)</Frame>
  </Tab>

  <Tab title="Management API">
    Pour activer la restriction liée à l’émetteur avec la Management API, envoyez une requête PATCH pour [mettre à jour le serveur de ressources](https://auth0.com/docs/api/management/v2/resource-servers/patch-resource-servers-by-id). Définissez les paramètres de l’objet `proof_of_possession` comme suit :

    <table class="table">
      <thead>
        <tr>
          <th><strong>Paramètre</strong></th>
          <th><strong>Description</strong></th>
        </tr>
      </thead>

      <tbody>
        <tr>
          <td><code>mechanism</code></td>
          <td>Définit la méthode de restriction liée à l’émetteur : <code>none</code>, <code>mtls</code> ou <code>dpop</code>.</td>
        </tr>

        <tr>
          <td><code>required</code></td>
          <td>Lorsqu’il est défini sur <code>true</code>, tous les jetons d’accès émis pour une application pour cette API seront restreints à cette application. Lorsqu’il est défini sur <code>false</code>, la restriction liée à l’émetteur n’est pas exigée pour l’application.</td>
        </tr>
      </tbody>
    </table>

    L’exemple de code suivant montre un corps de requête qui configure un serveur de ressources pour la restriction liée à l’émetteur avec mTLS :

    ```json lines
    "proof_of_possession": {
        "mechanism": "mtls",
        "required": true
      }
    ```
  </Tab>
</Tabs>
