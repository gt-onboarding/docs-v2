<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Le sender constraining des jetons à l’aide de Demonstrating Proof-of-Possession (DPoP) est actuellement en accès anticipé. Pour demander l’accès à cette fonctionnalité, communiquez avec votre représentant Auth0.
</Callout>

[Sender constraining](/docs/fr-CA/secure/sender-constraining) est un mécanisme de sécurité <Tooltip tip="" cta="View Glossary" href="/docs/fr-CA/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> et <Tooltip tip="" cta="View Glossary" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) qui lie de façon cryptographique les jetons d’accès et les <Tooltip tip="" cta="View Glossary" href="/docs/fr-CA/glossary?term=refresh+tokens">Jetons d’actualisation</Tooltip> à l’application cliente précise qui les a obtenus, ce qui empêche le vol et l’utilisation abusive des jetons.

Auth0 prend en charge le sender constraining mTLS et Demonstrating Proof-of-Possession (DPoP). Si vous activez le sender constraining pour une application cliente, vous devez aussi l’appliquer au <Tooltip tip="" cta="View Glossary" href="/docs/fr-CA/glossary?term=resource+server">serveur de ressources</Tooltip> auquel vous faites des appels d’API.

Pour configurer le sender constraining dans Auth0, vous devez :

* [Configurer le sender constraining pour une application cliente](#configure-sender-constraining-for-a-client-application).
* [Configurer le sender constraining pour un serveur de ressources](#configure-sender-constraining-for-a-resource-server).

<div id="how-it-works">
  ## Fonctionnement
</div>

Les <Tooltip tip="" cta="Afficher le glossaire" href="/docs/fr-CA/glossary?term=Access+tokens">jetons d&#39;accès</Tooltip> sont contraints à l&#39;expéditeur dans Auth0 selon la façon dont vous configurez la restriction à l&#39;expéditeur pour votre application cliente et votre serveur de ressources :

1. **Audience demandée :** Dans une requête de jeton, le fait que l&#39;audience demandée soit uniquement `/userinfo`, ou destinée uniquement à être utilisée avec le point de terminaison [`/userinfo`](https://auth0.com/docs/api/authentication/user-profile/get-user-info), ou qu&#39;il s&#39;agisse d&#39;une API personnalisée qui peut inclure `/userinfo` lorsque vous demandez aussi la portée openid, influe sur le fait que les jetons d&#39;accès soient contraints à l&#39;expéditeur.
2. **Application cliente :** Le fait que vous définissiez la restriction à l&#39;expéditeur comme `required` au niveau de l&#39;application cliente.
3. **Serveur de ressources :** Le fait que vous configuriez la restriction à l&#39;expéditeur pour le serveur de ressources :

   1. `none` : Vous n&#39;avez pas configuré de restriction à l&#39;expéditeur pour le serveur de ressources.
   2. `allowed` : Vous avez configuré la restriction à l&#39;expéditeur pour le serveur de ressources en définissant une méthode de restriction à l&#39;expéditeur.
   3. `required` : Vous avez configuré la restriction à l&#39;expéditeur comme obligatoire pour le serveur de ressources, ce qui signifie que les jetons d&#39;accès doivent être contraints à l&#39;expéditeur pour une application. Nécessite une méthode de restriction à l&#39;expéditeur.

* **Proof-of-Possession :** Le fait que l&#39;application cliente ait envoyé une assertion de preuve de possession (Proof-of-Possession) dans la requête de jeton :

  * Restriction à l&#39;expéditeur mTLS : La preuve de possession est démontrée par la présentation réussie, par le client, d&#39;une clé privée spécifique (associée à un certificat client) lors de la négociation TLS.
  * DPoP : La preuve de possession est obtenue lorsque le client crée un DPoP Proof JWT en le signant de manière cryptographique avec sa clé privée et en incluant le DPoP Proof JWT dans l&#39;en-tête HTTP DPoP de chaque requête où le jeton d&#39;accès associé est utilisé.

<div id="configure-sender-constraining-for-a-client-application">
  ## Configurer la restriction de l’émetteur pour une application cliente
</div>

Lorsque vous exigez une restriction de l’émetteur pour une application cliente, les jetons d’accès sont limités à cette application. Auth0 vérifie les requêtes pour s’assurer que seule l’application qui a demandé le jeton peut l’utiliser pour accéder à la ressource associée.

Une fois que vous avez configuré une application cliente pour exiger la restriction de l’émetteur, vous pouvez définir la méthode de restriction de l’émetteur, soit mTLS ou DPoP, lorsque vous configurez votre serveur de ressources.

Vous pouvez configurer la restriction de l’émetteur pour une application cliente avec le <Tooltip tip="" cta="Afficher le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> ou la <Tooltip tip="" cta="Afficher le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>.

<Tabs>
  <Tab title="Auth0 Dashboard">
    1. Accédez à [Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications). Sélectionnez l’application que vous voulez configurer.
    2. Sous **Settings**, faites défiler jusqu’à **Token Sender-Constraining**.
    3. Activez **Require Sender Constraining**. Désactivez cette option pour supprimer l’exigence de restriction de l’émetteur pour l’application.

    <Frame>![](/docs/images/cdy7uua7fh8z/3BoGW9NEYKPBkfk2tPBL7l/0f884ac3f64f17b23fd6a0e9436bb472/Screenshot_2024-07-23_at_1.30.19_PM.png)</Frame>
  </Tab>

  <Tab title="Management API">
    Pour configurer un client afin d’utiliser la restriction de l’émetteur, utilisez la [Management API](https://auth0.com/docs/api/management/v2).

    Pour exiger la restriction de l’émetteur pour un client, envoyez une requête PATCH pour [mettre à jour les paramètres du client](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id). Définissez le paramètre `require_proof_of_possession` sur `true`.

    ```json lines
    "require_proof_of_possession": true
    ```

    Pour supprimer l’exigence de restriction de l’émetteur, définissez le paramètre `require_proof_of_possession` sur `false`.

    ```json lines
    "require_proof_of_possession": false
    ```
  </Tab>
</Tabs>

<div id="configure-sender-constraining-for-a-resource-server">
  ## Configurer la contrainte de l’expéditeur pour un serveur de ressources
</div>

Les jetons d’accès émis par Auth0 peuvent être limités à l’expéditeur (c.-à-d. l’application cliente) qui doit accéder aux API d’un serveur de ressources.

Vous pouvez configurer la contrainte de l’expéditeur pour un serveur de ressources avec le Auth0 Dashboard ou la Management API.

<Tabs>
  <Tab title="Auth0 Dashboard">
    Pour activer la liaison de jeton (Token Binding) ou la contrainte de l’expéditeur, configurez les **API Settings** de votre API.

    1. Accédez à [Auth0 Dashboard &gt; Applications &gt; APIs](https://manage.auth0.com/#/apis).
    2. Sélectionnez l’API que vous voulez configurer.
    3. Sous l’onglet **Settings**, trouvez la section **Token Sender-Constraining**.
    4. Configurez ce qui suit :

       1. Sender Constraining Method :

          1. **None :** N’activez aucune méthode de contrainte de l’expéditeur pour votre serveur de ressources.
          2. **mTLS :** Activez mTLS comme méthode de contrainte de l’expéditeur pour votre serveur de ressources.
          3. **DPoP :** Activez DPoP comme méthode de contrainte de l’expéditeur pour votre serveur de ressources.

    B. Activez **Require Token Sender Constraining**. Tous les jetons d’accès émis à une application pour cette API seront limités à cette application.

    <Frame>![Auth0 Dashboard > APIs > Settings > Token binding](/docs/images/cdy7uua7fh8z/3Dv98iZosdpJMcZXfXyurn/823fa368ab874218fbb737aacac9b262/Screenshot_2025-07-28_at_3.59.01_PM.png)</Frame>
  </Tab>

  <Tab title="Management API">
    Pour activer la contrainte de l’expéditeur avec la Management API, envoyez une requête PATCH pour [mettre à jour le serveur de ressources](https://auth0.com/docs/api/management/v2/resource-servers/patch-resource-servers-by-id). Définissez les paramètres de l’objet `proof_of_possession` comme suit :

    <table class="table">
      <thead>
        <tr>
          <th><strong>Paramètre</strong></th>
          <th><strong>Description</strong></th>
        </tr>
      </thead>

      <tbody>
        <tr>
          <td><code>mechanism</code></td>
          <td>Définit la méthode de contrainte de l’expéditeur : <code>none</code>, <code>mtls</code> ou <code>dpop</code>.</td>
        </tr>

        <tr>
          <td><code>required</code></td>
          <td>Lorsqu’il est défini sur <code>true</code>, tous les jetons d’accès émis à une application pour cette API seront limités à cette application. Lorsqu’il est défini sur <code>false</code>, la contrainte de l’expéditeur n’est pas requise pour l’application.</td>
        </tr>
      </tbody>
    </table>

    L’exemple de code suivant montre un corps de requête qui configure un serveur de ressources pour la contrainte de l’expéditeur mTLS :

    ```json lines
    "proof_of_possession": {
        "mechanism": "mtls",
        "required": true
      }
    ```
  </Tab>
</Tabs>
