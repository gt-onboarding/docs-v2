  <Content>
    このガイドでは、`express-oauth2-jwt-bearer` パッケージを使用して、新規または既存の Express.js API アプリケーションに Auth0 を統合する方法を説明します。

    Auth0 Dashboard でまだ API を作成していない場合は、インタラクティブセレクターを使用して新しい Auth0 API を作成するか、既存のプロジェクト API を選択してください。

    Auth0 Dashboard で最初の API をセットアップするには、[入門ガイド](https://auth0.com/docs/get-started/auth0-overview/set-up-apis)を参照してください。各 Auth0 API は API Identifier を使用します。アプリケーションはこれを使用してアクセストークンを検証します。

    <Info>
      **Auth0 を初めて利用されますか？** [Auth0 の仕組み](https://auth0.com/docs/overview)について学び、
      OAuth 2.0 フレームワークを利用した [API の認証と認可の実装](https://auth0.com/docs/api-auth) についてもお読みください。
    </Info>

    <Section id={sections[0].id} title={sections[0].title} stepNumber="1" isSingleColumn>
      Permissions を使用すると、特定のアクセストークンを持つユーザーに代わって、リソースへどのようにアクセスできるかを定義できます。たとえば、ユーザーがマネージャーのアクセスレベルを持っている場合には `messages` リソースへの読み取りアクセスを許可し、管理者のアクセスレベルを持っている場合にはそのリソースへの書き込みアクセスを許可するといった設定が可能です。

      許可する Permissions は、Auth0 Dashboard の [APIs](https://manage.auth0.com/#/apis) セクションにある **Permissions** ビューで定義できます。

    <Section id={sections[2].id} title={sections[2].title} stepNumber="3" isSingleColumn>
      `express-oauth2-jwt-bearer` を、ドメインと API 識別子で設定します。

      右側に示されている `checkJwt` ミドルウェアは、リクエストに含まれるユーザーのアクセストークンが
      有効かどうかを確認します。トークンが有効でない場合、ユーザーがエンドポイントへアクセスしようとすると
      401 Authorization エラーが返されます。

      このミドルウェアは、要求されたリソースへアクセスするのに十分なスコープがトークンに含まれているかどうかは確認しません。

    <Section id={sections[3].id} title={sections[3].title} stepNumber="4" isSingleColumn>
      有効なJWTを必須とすることで個別のルートを保護するには、`express-oauth2-jwt-bearer`から構築した`checkJwt`ミドルウェアでルートを設定します。

      個々のルートを設定して、特定のスコープを確認できます。これを実現するには、`requiresScope`メソッドを使用して別のミドルウェアを設定します。必要なスコープを指定し、認可を追加したいルートにミドルウェアを適用してください。

      保護するルートに `checkJwt` と `requiredScopes` ミドルウェアを渡します。

      この設定では、`read:messages` スコープを持つアクセストークンのみがエンドポイントにアクセスできます。

      ### APIを呼び出す

      APIを呼び出すには、アクセストークンが必要です。テスト目的のアクセストークンは、[API設定](https://manage.auth0.com/#/apis)の**Test**ビューから取得できます。

      <Note>
        ##### チェックポイント

        アプリケーションの構成が完了したら、アプリケーションを実行して次の点を確認してください。

        * `GET /api/public` は、未認証のリクエストからアクセスできること。
        * `GET /api/private` は、認証済みリクエストからアクセスできること。
        * `GET /api/private-scoped` は、`read:messages` スコープを含むアクセストークンを持つ認証済みリクエストからアクセスできること。
      </Note>
