    このガイドでは、[Flask](https://flask.palletsprojects.com/)で構築された新規または既存のPython APIにAuth0を統合する方法を示します。

    Auth0 Dashboard でまだ API を作成していない場合は、インタラクティブセレクターを使用して新しい Auth0 API を作成するか、統合するプロジェクトに対応する既存の API を選択できます。

    または、[入門ガイド](https://auth0.com/docs/get-started/auth0-overview/set-up-apis)を読んで、Auth0 Dashboardで最初のAPIをセットアップすることもできます。

    Auth0のすべてのAPIは、API識別子を使用して設定されます。アプリケーションコードは、この識別子をオーディエンスとして使用してアクセストークンを検証します。

    <Info>
      **Auth0 を初めて利用しますか？** [Auth0 の仕組み](https://auth0.com/docs/overview) を学び、OAuth 2.0 フレームワークを使用した [API 認証と認可の実装](https://auth0.com/docs/api-auth) について確認してください。
    </Info>

      許可する Permissions は、Auth0 Dashboard の [APIs](https://manage.auth0.com/#/apis) セクションにある **Permissions** ビューで定義できます。

    <Section id={sections[2].id} title={sections[2].title} stepNumber="3" isSingleColumn>
      [Authlib](https://github.com/lepture/authlib) というライブラリを使って [ResourceProtector](https://docs.authlib.org/en/latest/flask/1/resource-server.html) を作成します。これは、指定したバリデーターでリソース（API ルート）を保護するための、[Flask のデコレーター](https://flask.palletsprojects.com/patterns/viewdecorators/)の一種です。

      このバリデーターは、リソースに渡されたアクセストークンについて、有効な署名とクレームを持っているかどうかを検証します。

      AuthLib の `JWTBearerTokenValidator` バリデーターを、いくつか調整を加えた上で使用し、[アクセストークンの検証](https://auth0.com/docs/secure/tokens/access-tokens/validate-access-tokens)に関する要件を満たすようにします。

      `Auth0JWTBearerTokenValidator` を作成するには、`domain` と
      `audience`（API Identifier）を渡す必要があります。これにより、トークンの署名検証に必要な公開鍵を取得し、
      それを `JWTBearerTokenValidator` クラスに渡します。

      続いて、このクラスの `claims_options` をオーバーライドし、トークンの有効期限、オーディエンス、および発行に関するクレームが、こちらの要件どおりに検証されるようにします。

    <Section id={sections[3].id} title={sections[3].title} stepNumber="4" isSingleColumn>
      次に、3つのAPIルートを持つFlaskアプリケーションを作成します:

      * `/api/public` は認証を必要としない公開エンドポイントです。
      * `/api/private` は、有効な JWT 形式のアクセストークンが必要なプライベートエンドポイントです。
      * `/api/private-scoped` 有効な JWT 形式のアクセストークンで、指定された `scope` を含むものを必要とするプライベートエンドポイントです。

      保護されたルートには `require_auth` デコレータが適用されます。これは、先ほど作成した `Auth0JWTBearerTokenValidator` を使用する `ResourceProtector` です。

      `Auth0JWTBearerTokenValidator`を作成するには、テナントのドメインと、先ほど作成したAPIのAPI識別子を渡します。

      `private_scoped` ルート上の `require_auth` デコレーターは、追加の引数
      `"read:messages"` を受け取り、先ほど作成したパーミッション(スコープ)がアクセストークンに含まれているかをチェックします。
